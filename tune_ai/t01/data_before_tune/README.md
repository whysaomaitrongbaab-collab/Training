## ① ใช้ Qwen ไหม? ตัวไหน?

**ใช่ — `Qwen3-VL-8B-Instruct`**

| ทำไม Qwen | |
|---|---|
| pipeline ที่มีอยู่ยิง Qwen-VL-Max อยู่แล้ว | prompt/schema/บทเรียนทั้งหมดผูกกับมันมา 3 อาทิตย์ |
| open-weight เทรนเองได้ | ไม่ต้องส่งแบบก่อสร้างลูกค้าขึ้น API เจ้าอื่น |
| Unsloth รองรับเต็ม | เทรนเร็วขึ้น 1.7× ใช้ VRAM น้อยลง 60% |

**ทำไม 8B ไม่ใช่ 4B / 32B:**

| ขนาด | VRAM (4-bit) | เหมาะกับ |
|---|---|---|
| 2B / 4B | 4-8 GB | caption/tag ง่าย ๆ — **อ่านเอกสารหนาแน่นสู้ 8B ไม่ได้** |
| **8B** ✅ | **~6 GB weights** | เอกสารหนาแน่น + ตาราง + structured extraction — ตรงงานเรา |
| 32B | ~21 GB | แม่นกว่า แต่ 24GB เทรนไม่ไหว ต้อง A100 80GB ค่าเช่า 5-10× |

งานเรา = อ่านตัวเลขเล็กในแบบก่อสร้าง + คืน JSON โครงสร้างซับซ้อน → 8B คือจุดคุ้มสุด

## ③ ต้องทำยังไง (ทำตามนี้ทีละข้อ)

```bash
# 1) เช่าเครื่อง Vast.ai — เลือก image ที่มี PyTorch + CUDA 12.x, disk 80GB, เปิด Jupyter หรือ SSH

# 2) อัปโหลดโฟลเดอร์นี้ทั้งก้อน (~240 MB) ขึ้น /workspace/
#    ถ้าเน็ตช้า: zip ก่อนแล้วค่อยอัป หรือ scp -r

# 3) ติดตั้ง (ครั้งเดียว)
pip install unsloth trl peft accelerate bitsandbytes
pip install qwen-vl-utils pillow

# 4) วัด baseline ก่อน! (โมเดลยังไม่ทูน เก่งแค่ไหน — ไม่มีตัวเลขนี้จะไม่รู้ว่าเทรนแล้วดีขึ้นจริงไหม)
python eval_fields.py --base --limit 20

# 5) เทรน (~1.5-2.5 ชม.)
python train_qwen3vl.py

# 6) วัดผลหลังเทรน แล้วเทียบกับข้อ 4
python eval_fields.py
```

**ดูอะไรตอนเทรน:** ถ้า VRAM เต็ม (OOM) → ลด `MAX_PIXELS` ใน `train_qwen3vl.py` เป็น `4096*32*32` ก่อน
อย่าลด `BATCH` (เป็น 1 อยู่แล้ว) และอย่าปิด `finetune_vision_layers`

**ตัวเลขที่ควรได้:** ยังไม่มีใครวัด — ข้อ 4 คือ baseline ตัวแรกของโปรเจกต์นี้
ที่ต้องดูคือ **element recall** (โมเดลมองข้ามคานไหม) ไม่ใช่ eval loss

## การตัดสินใจที่ฝังอยู่ในชุดนี้ (มะขามสั่งเปลี่ยนได้ทุกข้อ)

### 1 ภาพ = 1 example, รวมทุก view
310 ภาพ → 310 examples (+ 4 gridline) ไม่ใช่ 474 ตาม 1 ไฟล์ = 1 example
เพราะสเปค §3 บอกเองว่างานคือ *"inventory every view first"* แล้วค่อยแยกไฟล์ — การแยกไฟล์เป็นวิธี**เก็บ** ไม่ใช่ขอบเขตของงาน
ถ้าเทรนแยก view ต้องบอกโมเดลว่า "เอา view ไหน" ซึ่งตอน inference เราไม่มีข้อมูลนั้น = สร้างบั๊กบทเรียนข้อ 7 ขึ้นมาใหม่
ทุกหน้าใช้ `views[]` เหมือนกันหมด (หน้า view เดียวก็เป็น array ยาว 1) เพื่อให้รูปทรง output คงที่

### ⚠️ ตัด `specs{}` ออก — จุดนี้สำคัญที่สุด
`specs{}` ทุกบล็อกในชุดนี้มี `spec_source` ชี้ไป**หน้าอื่น** เช่น
`หน้า33_beam_plan` (บ้าน04) เอา specs มาจาก `หน้า37_beam_slab_sections`

แปลว่าขนาด/จำนวนเหล็กพวกนั้น **ไม่ได้พิมพ์อยู่บนภาพหน้า plan เลย** ถ้าเทรนให้ตอบ = สอนโมเดลมโนตัวเลขเหล็กตรง ๆ
ซึ่งเป็นข้อผิดพลาดที่ CLAUDE.md บทเรียนข้อ 2 บอกว่า VLM พลาดบ่อยที่สุดอยู่แล้ว (Ø12 → "DB23")
การ join spec เข้ากับ position เป็นงานของ**โค้ดปลายทาง** ตามสเปค §7 ไม่ใช่ของโมเดล

กระทบ 10 examples · เปิดคืนได้ที่ `CFG.DROP_CROSS_PAGE_SPECS = false`

### ตัดบันทึกการทำงานออกจาก target
ตัด `source_image`/`source_pages`/`grid_source`/`sibling_files` (โมเดลเห็นแค่ภาพ ไม่มีทางรู้ path)
ตัด `warnings`/`note`/`*_note`/`source` (เป็นบันทึกรีวิว ไม่ใช่ข้อมูลจากแบบ — 1,577 รายการ เฉลี่ย 262 ตัวอักษร)
กรอง `confidence_flags`/`spec_confidence_flags` เอาเฉพาะที่ไม่มีวันที่/ชื่อคน/รอบ t1-t4

ไม่งั้นโมเดลจะเรียน**แต่งวันที่กับชื่อคนมั่ว** · `build_dataset.js` มีตัวตรวจรั่วยิงทุกครั้งที่ build
ปิดได้ที่ `CFG.STRIP_WORKLOG = false`

### instruction สั้น (284 tok) ไม่ยัดสเปคเต็ม
fine-tune คือการฝัง format ลงน้ำหนักโมเดล ถ้ายัดสเปค 6,000 tok ทุก example = เปลืองฟรี
**ต้องใช้ instruction ตัวเดียวกันเป๊ะตอน inference** (อยู่ใน `PROMPT_SHORT` ของ `build_dataset.js`)
สลับเป็นสเปคเต็มได้ที่ `CFG.PROMPT_MODE = 'full_schema'`

### แบ่ง val ตาม "หลัง" ไม่ใช่สุ่มตามหน้า
หน้าในบ้านเดียวกันใช้กริด/มาร์ค/ลายมือชุดเดียวกัน — สุ่มตามหน้าแล้วตัวเลขจะสวยเกินจริง (data leakage)
val = บ้าน 04 เพราะตรวจละเอียดที่สุด (label val ผิด = ตัวเลขวัดโกหก)
เปลี่ยนได้ที่ `CFG.VAL_HOUSES`

### ทำไม 5,120 visual tokens
ทดสอบย่อภาพจริงแล้ว crop จุดเดียวกันมาดู (บ้าน04 หน้า37):

| max tok | ขนาดเต็มหน้า | `Ø9มม.@0.15` | จุดเหล็กบน/ล่าง |
|---|---|---|---|
| native 7,558 | 3309×2339 | ชัด | แยก 2 วง ✓ |
| **5,120** ✅ | 2723×1925 | ชัด | **แยก 2 วง ✓** |
| 2,560 | 1926×1361 | "มม." เละ | **รวมเป็นก้อน ✗** |
| 1,280 | 1362×963 | เพี้ยน "Ø91บ." | **รวม ✗** |

สเปค §6 บังคับแยก `main_bar.top`/`bottom` เสมอ — ถ้าจุดเหล็กรวมกันตั้งแต่ input โมเดลไม่มีทางตอบถูก

⚠️ **Qwen3-VL นับ 1 token ต่อ 32×32 px ไม่ใช่ 28×28 แบบ Qwen2.5-VL** — ก็อป config เก่ามาจะ mis-size เงียบ ๆ แล้ว OOM

## ⚠️ ข้อจำกัดที่ต้องรู้ก่อนเชื่อผล

**83% ของ target ยังไม่มีคนตรวจ** — diff เทียบ `raw_json_ตัวที่ใช้งานจริง/` แล้วมีแค่ **80 จาก 474 ไฟล์** ที่แก้จริง อีก 394 ไฟล์เหมือน raw เป๊ะ

| pattern | ไฟล์ | ตรวจแล้ว |
|---|---|---|
| `material_list` | 159 | **0** |
| `plan` | 73 | 42 |
| `section` | 79 | 9 |
| `notes`/`index`/`symbol`/`unknown`/`title` | 62 | **0** |

งาน 3 อาทิตย์ลงที่ `plan` + grid เป็นหลัก — ถูกต้องตามที่ตั้งใจ แต่แปลว่าโมเดลจะเรียน**ผลสกัดที่ยังไม่มีใครตรวจ**เป็นส่วนใหญ่ รวมถึงข้อผิดพลาดเดิมของมันเอง

**ไม่ได้แปลว่าอย่าเทรน** — รอบแรกคือการหา baseline ว่าโครงสร้างทั้งหมดใช้ได้ไหม
แต่เวลาอ่านผล ให้ดู `plan`/`section` เป็นหลัก ส่วน `material_list` ถือว่ายังเชื่อไม่ได้จนกว่าจะมีคนตรวจ
