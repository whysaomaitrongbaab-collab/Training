**งานที่ทำ (เรียงตามลำดับ):**

### 1. เพิ่ม dummy grid ที่พลาดไปก่อนหน้า
- **`D'` = 2.8** (แกน y, ระหว่างกริด D-C) — ผนังกั้นเต็มความกว้างระหว่างแถวห้องนอนกับแถวห้องด้านหลัง (หน้า06) — cascade แก้ `หน้า06_floor_plan.json` (นอน1/นอน2 เปลี่ยนจาก `D-C` เป็น `D-D'`) แต่**จับผิดตัวเองได้ก่อนส่ง**: เกือบแก้ ห้องน้ำ_WC1 เป็น `D'-C` ตามการเดาต่อเนื่อง แต่เช็ค `หน้า11_wc1_plan.json` (แบบขยายเฉพาะ) ก่อนแล้วพบว่าระบุตรงๆ ว่าอยู่ `between gridB and gridC` — กลับไปใช้ `C-B` เดิม ไม่ใช้ค่าที่เดาไว้ (บทเรียน: เช็ค detail sheet ก่อน cascade เสมอ)
- **`1''` = 3.45** (แกน x, ช่องเดียวกับ `1'`) — มะขามสั่งเพิ่มตรงๆ ยังไม่ได้ pinpoint เส้น dimension ที่แน่ชัด 100% ด้วยตัวเอง ยังไม่ cascade ไปหน้าอื่นเพราะไม่มีหลักฐานชัด (รอคำสั่ง/หลักฐานเพิ่ม)
- ทั้งสองใส่ `confidence_score: 1` ตาม convention (ค่าที่มะขามยืนยันเอง)

### 2. Convention ใหม่: แกนตั้ง (row) มาก่อนแกนนอน (column) ใน `grid_ref` แบบ range
พบว่าตอนย่อ `grid_ref` ของหน้า06 รอบแรกเขียนแกนนอนก่อนโดยไม่ตั้งใจ (`"1-1' x D-C"`) มะขามสั่งกลับเป็นแกนตั้งก่อน (`"D-C x 1-1'"`) — ตรงกับ convention เดิมของ `"A-1"` (row ก่อน column) อยู่แล้ว เพิ่มเป็นกฎลายลักษณ์อักษรใน `primary_rawjson_schema.md` section 4 ด้วย (อ้างอิง `rule_of_tune.md` กฎข้อ 7 ก่อนแก้)

### 3. Convention ใหม่: point-type `grid_ref` ไม่ใช้ขีด (dash) เก็บขีดไว้แค่ line/area range
มะขามชี้ว่า footing/column (`grid_refs: ["C-2"]`) เป็นแค่ "จุด" ไม่ใช่ "เส้น/พื้นที่" ที่ต้องมีขีดคั่น — สั่งเปลี่ยนเป็น `"C2"` แล้วขยายไปถึงคาน (`grid_ref_start`/`grid_ref_end`) ด้วย เพราะแต่ละ field ก็เป็นจุดปลายคานจุดเดียว ไม่ใช่ตัวช่วงเอง (ช่วง/เส้นคานแสดงด้วยการมี 2 field แยกอยู่แล้ว) — แก้ `หน้า19_view1_footing_plan.json` และ `หน้า19_view2_beam_plan.json` ครบ (ใช้ regex script ตัดขีดระหว่างแถวตัวอักษร-คอลัมน์ตัวเลขในทุก grid_ref_start/end แล้ว verify ด้วย grep) **ยังไม่ได้แก้ `primary_rawjson_schema.md`** เพราะ section 4 ปัจจุบันเขียนตรงข้าม (บอกว่า point-type ใช้ `"A-1"` มีขีด) — ถามมะขามว่าจะเปลี่ยนทั้งโปรเจกต์หรือแค่บ้าน 1 ก่อน ยังไม่ได้คำตอบชัดตอนเขียน entry นี้

### 4. ลบ element ที่ไม่เกี่ยวข้อง + ลบ grid ที่ซ้ำซ้อน
- ลบ `doors_and_windows_summary` element ออกจาก `หน้า06_floor_plan.json` ตามคำสั่ง
- พบว่า `หน้า19_view1_footing_plan.json` มีทั้ง `grid_source` (ชี้ไป grid master ถูกต้อง) **และ** inline `"grid": {...}` ซ้ำซ้อน (ขัดกับสเปคเองที่บอกว่า "never re-embed the full grid inline inside every view") — มะขามจับได้ ลบ inline grid ออก เช็คไฟล์อื่นในบ้านแล้วไม่มีที่ไหนซ้ำแบบนี้อีก

### 5. บันทึกบทเรียน "อ้างกฎแล้วไม่หยุดถามจริง" ลง `rule_of_tune.md` (กฎข้อ 9 ใหม่)

### 7. `หน้า19_view2_beam_plan.json` — เรียงลำดับ elements[] ใหม่ตามการอ่านแบบ + แยก spec ซ้ำออกเป็น `specs{}`
มะขามสั่งให้เรียง element ในไฟล์ตามลำดับการอ่านจริง (บนลงล่าง, ซ้ายไปขวา, แนวนอนก่อนแนวตั้งถ้าจุดเริ่มตรงกัน) แทนการกลุ่มตาม element_id/mark แบบเดิม — เขียนใหม่ทั้ง 20 elements ตามลำดับตำแหน่งจริง (verify ครบไม่มีข้อมูลหาย) แล้วเพิ่มเป็นกฎทั่วไปใน `primary_rawjson_schema.md` section 4 ("Element ordering within elements[]") — มะขามยืนยัน "yes" ให้เพิ่ม (คราวนี้หยุดถามจริงก่อนแก้ตามกฎข้อ 9 ที่เพิ่งตั้ง)

จากนั้นมะขามชี้ต่อว่าทุก occurrence ของคานมาร์คเดียวกันพิมพ์ spec (`width_mm/height_mm/main_bar/stirrup/additional_bars/concrete_grade/steel_grade/spec_source/spec_confidence_score`) ซ้ำเหมือนกันทุกตัวอักษร กลายเป็น "god object" — สั่งแยกออกมาเป็น object ใหม่ `specs{}` keyed by `element_id` (ตรวจสอบก่อนว่าทุก occurrence ของแต่ละมาร์คมี spec เหมือนกันเป๊ะจริง: B2 x2, B4 x3, B5 x3, B3 x3, B4X x2 ยืนยันแล้ว) `elements[]` เหลือแค่ตำแหน่ง+confidence เฉพาะจุด — **ถามมะขามต่อว่าจะเพิ่ม pattern `specs{}` นี้ใน `primary_rawjson_schema.md` ด้วยไหม ยังไม่ได้คำตอบ** มะขามบอกเหนื่อยแล้วหยุดตรงนี้
