### 4. มะขามเริ่มสร้างโปรเจกต์จริงใน Label Studio Cloud — เจอบั๊ก scroll จริง
มะขามสร้างโปรเจกต์ Elements สำเร็จ (project 271482) แล้วส่ง screenshot มาว่า **"tab ซ้ายเลื่อนขึ้นลงไม่ได้แล้วจะดูแปลนยังไง"** — วินิจฉัยพบว่า `.stickycol` (คอลัมน์ปักรูป+header) ไม่มี `max-height`/`overflow-y:auto` พอเนื้อหาสูงกว่าจอ ส่วนเกินโดนตัดหายไปเฉยๆ มองไม่เห็นเลย ไม่ใช่แค่ "เลื่อนไม่ได้" แต่ "เนื้อหาหายไปจริง"

**แก้:** เพิ่ม `max-height:96vh; overflow-y:auto; padding-right:6px;` ให้ `.stickycol` ใน **ทั้ง 5 ไฟล์** (elements/elements-freezeR/single/single-freezeR/material_list)

### 5. ลบ/ถอน browser "no style information" fix เพราะทำให้แย่ลง
ระหว่างแก้ข้อ 4 มะขามขอให้เอาข้อความเตือน "This XML file does not appear to have any style information..." ออกด้วย (ข้อความมาตรฐานของ browser ตอนเปิดไฟล์ `.xml` ตรงๆ) — เพิ่ม `<?xml-stylesheet type="text/css" href="data:text/css,*{}"?>` ไปบนสุดทั้ง 5 ไฟล์

**ผลข้างเคียงที่ไม่คาดคิด:** พอมะขามเปิดไฟล์ตรงๆ ใน tab ใหม่ กลายเป็นโชว์ข้อความมั่วๆ (CSS ของ `<Style>` และข้อความใน `<Choice>` หลุดออกมาปนกัน ไม่มีโครงสร้าง) เพราะ browser เปลี่ยนโหมด render พอเจอ stylesheet (แม้จะว่างเปล่า) — มะขามงงและหงุดหงิด ("อะไรวะ") → อธิบายสาเหตุแล้ว **ถอนการแก้ทั้งหมดออก** (ลบบรรทัด `<?xml-stylesheet?>` ออกจากทั้ง 5 ไฟล์) และย้ำว่าไฟล์พวกนี้ไม่ได้ถูกออกแบบให้เปิดตรงๆ ใน browser เลย ต้อง paste เข้า Label Studio Code editor เท่านั้นถึงจะเห็นผลถูกต้อง

6. **เจอปัญหาที่ 2 ระหว่างตรวจสอบขั้นสุดท้าย — จุดสำคัญที่สุดของ entry นี้:**
   ตรวจสอบ `.gitignore` เพื่อยืนยันว่าไม่ได้ถูกแก้ (เพราะระบบเคยแจ้งว่า "rejected" ไปแล้วในข้อ 3) — **กลับพบว่าเนื้อหาไฟล์ถูกแก้เป็นชื่อใหม่จริง ทั้งที่ระบบรายงานว่า tool call นั้นถูกปฏิเสธ** — คือ**ข้อความ "rejected" ไม่ตรงกับสถานะไฟล์บนดิสก์จริง** เป็นความคลาดเคลื่อนระหว่าง harness กับผลลัพธ์จริงที่เกิดขึ้น — แก้ไขซ้ำอีกรอบให้ `.gitignore` กลับเป็น `raw/` ตามเดิม

### ผลลัพธ์สุดท้าย (ยืนยันแล้ว)
- `raw/` กลับมาชื่อเดิม เนื้อหาครบ (README.md, image, คำตอบของQwenเพียว ๆ, แบบบ้าน) ไม่มีไฟล์หาย
- 5 script กลับไปอ้างอิง `'raw'` ครบทุกจุด (grep ยืนยันแล้ว)
- `.gitignore` กลับไปอ้างอิง `raw/` ครบ (grep หาคำว่า "ข้อมูลเก่าที่ไม่ใช้แล้ว" เหลือ 0 จุดในทั้ง repo)
- `git status` สุดท้ายสะอาด ไม่มี diff ค้างจากเหตุการณ์นี้เลย

### บทเรียน / ข้อควรระวังสำหรับครั้งต่อไป
1. **ก่อน rename/ย้ายโฟลเดอร์ข้อมูลเสมอ ต้อง grep หา path ที่ hardcode อยู่ในโค้ดทุกไฟล์ก่อน** ไม่ใช่แค่เดาจากชื่อโฟลเดอร์ว่า "เก่า/ไม่ใช้แล้ว" — ชื่อที่ user เข้าใจว่าเก่า อาจไม่ตรงกับความจริงในโค้ด (raw/ ดูเหมือนชื่อเรียบง่ายแต่จริงๆ เป็น path หลักของ pipeline)
2. **Windows "Access denied" ตอน rename/delete โฟลเดอร์ ไม่ได้แปลว่าไฟล์เสีย** — มักเกิดจากโปรแกรมอื่น (File Explorer, terminal, background scan) เปิดโฟลเดอร์นั้นค้างอยู่เป็น working directory ให้ลองปิดหน้าต่างที่เกี่ยวข้องแล้ว retry ก่อนตื่นตระหนกว่าข้อมูลพัง
3. **ข้อความ "tool use rejected" จาก harness ไม่ควรเชื่อ 100% ว่าไฟล์ไม่ถูกแก้จริง** — ต้อง verify สถานะไฟล์บนดิสก์จริงเสมอหลังเจอ rejection โดยเฉพาะก่อนสรุปว่า "ไม่ได้แก้อะไรไปแล้ว" — เคสนี้เกือบทำให้ `.gitignore` ค้างเป็นชื่อผิดถ้าไม่ได้ตรวจซ้ำตอนท้าย
4. งานนี้จบแบบ **undo กลับสู่สภาพเดิม 100%** ไม่มีความเสียหายต่อข้อมูลจริงหลงเหลืออยู่ — บันทึกไว้เผื่อวันหลังสงสัยว่าทำไม git log/diff ถึงมีร่องรอยการแก้ไฟล์พวกนี้ไปมา
