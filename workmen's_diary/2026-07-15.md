## [ช่วงดึก/เช้ามืด] มะขาม (ทำงานร่วมกับ Claude Code — ต่อเนื่องจากรีวิวบ้านหลังที่ 1, ดู `2026-07-14.md`)

**บริบท:** ต่อจากจุดที่ "หยุดตรงนี้" ใน `2026-07-14.md` (มะขามพักแป๊บ) กลับมาทำต่อข้ามวัน ไล่รีวิว หน้า19 (แก้ต่อ), หน้า20, หน้า21, หน้า22, หน้า60-61 จนจบ ก่อนจะย้ายไปเริ่มบ้านหลังถัดไป (หลังที่ 6)

**งานที่ทำ:**

### 1. Element ordering แก้กลับทิศทาง: **แนวตั้งมาก่อนแนวนอน** (ที่เขียนไว้ตอนแรกผิด)
มะขามชี้ว่ากฎ element ordering ที่เพิ่งตั้งเมื่อวาน เขียนทิศผิด — ที่ถูกคือแนวตั้งมาก่อน (เช่น D1: `B4 D1-C1` ก่อน `B2 D1-D2`) เรียง `หน้า19_view2_beam_plan.json` ใหม่ทั้งไฟล์ตามนี้ (validate 20 elements ครบ) แก้ `primary_rawjson_schema.md` กลับทิศตามแล้ว (ยืนยัน "y" ก่อนแก้) ระหว่างทางเจอ+แก้ B2 ที่พิมพ์ผิดเป็น `D'2/D'3` กลับเป็น `D2/D3` ด้วย

### 2. เพิ่มคานที่หายไปอีก 7 เส้น + แก้ B4X 3'/3'' กลับให้ถูก
มะขามส่งภาพเต็มหน้า19 (ชัดกว่าเดิม) ชี้คานที่หายไป 7 เส้น: `B2 D'2-D'3` (ยืนยันว่า D' เป็นคานจริงฝั่งขวาด้วย ไม่ใช่แค่ผนัง), `B4 B1-B2`/`B4 B2-B3` (คานทับหลัง row B คู่กับ row C), `B2 B1''-A1''` (แก้ปริศนาจุดที่หาไม่เจอเมื่อก่อน), `B2 C3'-B3'`, `B3 A1''-A3`/`B3 A'1''-A'3` (คานทับหลัง row A/A') — เพิ่มครบ confidence 1 ทุกตัว รวม 27 elements

พบ B4X สองช่วงถูกแก้ label เพี้ยนกลับเป็น `3'` (span ไม่ตรงกันเอง 0.6 vs 1.5) ทั้งที่ grid master บันทึกไว้แล้วว่าทั้งคู่ควรเป็น `3''` (1.5 ทั้งคู่) — ยืนยันจาก grid master (3-3'=0.6 จริง) แล้วแก้กลับ

### 3. Sync หน้า20 + หน้า21 ให้ตรงกับหน้า19 (mark/spec เดียวกัน)
- `หน้า20_view1_tie_beam_plan.json` (B2×15, B6×2): ตัดขีด, เรียงลำดับใหม่, แยก `specs{}`, รวม additional_bars B2 เข้า bottom (2→3) เหมือนกัน
- `หน้า20_view2_roof_frame_plan.json`: ตัดขีดใน grid_refs อย่างเดียว
- `หน้า21_view1_section.json` (ต้นทาง spec ทุกคาน): sync การ merge ทั้งหมดจากหน้า19 — พบ 1 จุดขัดแย้งจริง (B4 อ่านจากภาพนี้ได้ additional bar 1 เส้น แต่หน้า19 มี 2) ใส่ flag ให้คนตรวจ ไม่เลือกฝ่ายใดเงียบๆ
- `หน้า21_view2_schedule.json` (ตารางเสา C1): **มะขามจับได้ปัญหาคนละแบบ** — เสาใช้โครงสร้าง `main_bar.top/bottom` แบบคาน (top4+bottom4=รวม 8 ผิด) ทั้งที่เสาไม่มี "หน้าบน-ล่าง" มีแค่เหล็กรอบเสา 4 มุมเส้นเดียว (4-Ø12มม.) — แก้เป็น `main_bar: {count:4}` เดี่ยว ยังไม่ถามว่าจะเพิ่มเป็นกฎทั่วไปไหม

### 4. หน้า22 — ปฏิเสธคำแนะนำผิดจาก AI ภายนอก
มะขามส่งคำวิเคราะห์จาก AI ตัวอื่นที่เสนอให้แก้ F2 main bar จาก RB/SR24 เป็น DB/SD30 (อ้างว่า 12mm มักเป็น DB) — **Claude เช็คภาพต้นฉบับเองก่อนแล้วปฏิเสธ** เพราะสัญลักษณ์ที่พิมพ์จริง (Ø) เหมือนกันทั้ง F1(Ø9)/F2(Ø12) ต่างแค่ขนาด ตรงกับกฎ "Ø เสมอ=RB ห้ามเดาจากขนาด" ที่มีอยู่แล้ว ไม่แก้ไฟล์เพราะข้อมูลเดิมถูกอยู่แล้ว — **บทเรียน: AI ภายนอกให้คำแนะนำผิดได้ ต้องเช็คภาพเองก่อนเชื่อทุกครั้ง**

### 5. หน้า60-61 — เพิ่ม pattern ใหม่ `misc` (เบ็ดเตล็ด)
มะขามสั่งเปลี่ยน pattern จาก `title` เป็นคำอังกฤษของ "เบ็ดเตล็ด" สำหรับ 2 หน้านี้ (หน้า60=ตารางเปรียบเทียบราคา 10 แบบบ้าน, หน้า61=ปกคอลลาจรวมทุกแบบ) เพราะเป็นเนื้อหาโปรโมทรวมทั้งซีรีส์ ไม่ใช่ปก/ข้อมูลของบ้านหลังนี้เอง — เพิ่ม pattern ที่ 14 `misc` ใน `primary_rawjson_schema.md` (taxonomy จาก 13→14 ประเภท) ตามคำสั่งตรงๆ ในข้อความเดียว (ไม่ต้องถามซ้ำเพราะชัดเจนแล้ว)

**ไฟล์ที่เปลี่ยนวันนี้:** `หน้า19_view2_beam_plan.json`, `หน้า20_view1_tie_beam_plan.json`, `หน้า20_view2_roof_frame_plan.json`, `หน้า21_view1_section.json`, `หน้า21_view2_schedule.json`, `หน้า60.json`, `หน้า61.json`, `primary_rawjson_schema.md` (×3 จุด), `primary_rawjson_schema_edit_log.md`, `สิ่งที่ต้องแก้.md` (ข้อ 14-17)

**สรุปก่อนขึ้นบ้านหลังที่ 6 — convention ที่ยืนยันแล้วและควรใช้กับบ้านถัดไปตั้งแต่แรก:**
1. Point-ref ไม่มีขีด (`C2` ไม่ใช่ `C-2`) — ขีดสงวนไว้แค่ line/area range (`D-C`)
2. แกนตั้ง (row) มาก่อนแกนนอน (column) เสมอในการเขียน range
3. `elements[]` เรียงตามตำแหน่งจริง: บนลงล่าง, ซ้ายไปขวา, **แนวตั้งก่อนแนวนอน**เมื่อจุดเริ่มตรงกัน
4. Spec ที่ซ้ำกันทุก occurrence ของมาร์คเดียวกัน → แยกเป็น `specs{}` keyed by element_id ไม่พิมพ์ซ้ำใน elements[]
5. `additional_bars[].position` ต้องเช็คกับ leader line จริงเสมอ ห้ามเชื่อคำที่พิมพ์ไว้ — merge เข้า main_bar (บนหรือล่าง แล้วแต่หลักฐาน) ถ้าเป็นเหล็กเสริม/ล้วงเข้าคานเดียวกันหรือคานข้างเคียง
6. เสา (column) ไม่ใช้ `main_bar.top/bottom` แบบคาน — ใช้ `main_bar.count` เดี่ยว (ยังไม่ได้เพิ่มเข้า schema อย่างเป็นทางการ)
7. Pattern `misc` (เบ็ดเตล็ด) สำหรับหน้าโปรโมท/อ้างอิงรวมทั้งซีรีส์ (ไม่ใช่ของบ้านหลังนี้เอง) แยกจาก `title` (ปกของบ้านหลังนี้)
8. ทุกครั้งที่แก้ `primary_rawjson_schema.md` ต้องอ้างอิง `rule_of_tune.md` กฎข้อ 7, หยุดถาม yes/no จริงตามกฎข้อ 9-10 (ห้ามเดาคำตอบจากข้อความสั้น/กำกวม), แล้ว log ใน `primary_rawjson_schema_edit_log.md` เสมอ

**⚠️ ยังไม่ตอบ/ยังไม่ทำก่อนขึ้นบ้าน 6:**
- Scope ของ point-ref convention (ทั้งโปรเจกต์ vs บ้าน 1 อย่างเดียว) — ยังไม่ยืนยันชัด
- เพิ่มกฎเสาไม่ใช้ top/bottom เข้า schema อย่างเป็นทางการไหม
- ไฟล์ที่เหลือในบ้าน 1 (นอกเหนือจาก หน้า00/05/06/06_roof/11/19×2/20×2/21×2/22/60/61) ยังไม่ได้ไล่แก้ตาม convention ใหม่ทั้งหมด
- Sync กลับ `raw_json_ตัวที่ใช้งานจริง/01บ้าน_เล็ก_1ชั้น_01/` ยังไม่เริ่มเลย

## [ช่วงบ่าย/เย็น] มะขาม (ทำงานร่วมกับ Claude Code — ย้ายไปเริ่มบ้านหลังที่ 2 แทนหลังที่ 6)

**บริบท:** มะขามเปลี่ยนแผน ให้เริ่มรีวิว `json_แก้ไขแล้ว/02บ้าน_เล็ก_1ชั้น_02/` (แทนที่จะไปบ้านหลังที่ 6 ตามที่วางแผนไว้ท้าย entry ก่อนหน้า) — คำสั่ง 2 ข้อ: (1) apply สิ่งที่ต้องแก้.md's convention ทั้งหมดกับบ้าน 2, (2) รีเช็คเฉพาะไฟล์ pattern=plan (15 ไฟล์) กับภาพต้นฉบับ

**งานที่ทำ:**

### 1. Grid_ref format convention ไล่แก้ 12 ไฟล์
ตัด `grid`/`dummy` prefix, แกนตั้งมาก่อนแกนนอน, `~` แทน near/approx, point-ref ตัดขีด — หน้า06,07,10(×2),12,14,16,17,23,24,39_floor,40 ตรวจกับภาพต้นฉบับทุกไฟล์ pattern=plan ครบ (ห้อง/footing/column ทุกจุดถูกต้องตรงภาพ) ระหว่างทางเจอว่า หน้า07/หน้า10_section2 ต้นฉบับเขียนลำดับแถว D-E/E-F **กลับทิศอยู่แล้ว** (ต้องเป็น E-D/F-E ตามลำดับบนลงล่าง F>E>D>C>B>A) แก้ให้ถูกไปพร้อมกัน

### 2. พบบั๊ก additional_bars position ผิดฝั่งซ้ำ (เหมือนบ้าน 1 เป๊ะ) — หน้า25/28
ซูมภาพหน้าตัดคาน B2/B2X ใน S-07 (หน้า28_beam_slab_sections.json) นับจุดเหล็กในหน้าตัดจริงแทนที่จะเชื่อ label ที่บันทึกไว้: B2's "หยุดที่ L/8" ที่เคยระบุ "บนคาน" จริงๆ อยู่ล่าง (จุดรวมกับเหล็กล่างหลัก 4 จุด) แก้เป็น bottom 2→4; B2X's lap-splice ที่เคยระบุ "กลางคาน" จริงๆ อยู่บน (3 จุดรวมที่บน) แก้เป็น top 2→3 — cascade เข้า `หน้า25_beam_plan.json`'s specs{} ที่แยกออกมาใหม่พร้อมกัน (รวม elements[] reorder vertical-before-horizontal, 20 elements)

### 3. พบ mislabel จริงใน หน้า26_tie_beam_plan.json (RB1A ที่แท้คือ RB1)
ซูมภาพ S-05 บริเวณห้องเล็กใต้กริด C: เสาแนวตั้ง col3 (C3-B3) พิมพ์ label "RB1" จริงๆ (ไม่ใช่ RB1A ตามที่บันทึกไว้) — RB1A เริ่มจริงแค่ใต้กริด B ลงไป แก้ element_id+spec กลับ ทำให้ห้องสมมาตรกับฝั่ง col2 (ตรงกับหน้า25) — ทำ specs{} extraction + reorder ให้ไฟล์นี้ด้วย (28 elements)

### 4. หน้า29_column_schedule — ยืนยันกฎเสาใช้ main_bar.count เดี่ยว (ตอบคำถามค้างจากบ้าน 1)
ข้อมูลเดิมใช้โครงสร้าง top/bottom แบบคาน (มี flag เตือนตัวเองว่ารวมจริงคือ 4 ไม่ใช่ 8) แก้เป็น `main_bar:{count:4,...}` เดี่ยวตามที่ตรวจกับภาพ (4-Ø12มม. รอบเสา) — **นี่คือคำตอบของคำถามที่ค้างจากบ้าน 1: ยืนยันแล้วว่าเสาไม่ใช้ top/bottom ทั่วโปรเจกต์**

### 5. เพิ่ม pattern misc ให้หน้า71/72 (ตารางราคา + ปกคอลลาจรวมซีรีส์)
ตรงกับ precedent บ้าน 1 หน้า60/61 เป๊ะ — หน้า71 เดิมเป็น `unknown` เปลี่ยนเป็น `misc`, หน้า72 เดิมเป็น `title` เปลี่ยนเป็น `misc`
