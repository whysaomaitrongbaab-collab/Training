# สิ่งที่ต้องแก้

บันทึกละเอียดของทุกคำสั่งแก้ไขที่มะขามสั่งใน `json_แก้ไขแล้ว\` — เขียนให้ละเอียดพอที่จะ**เอาไปใช้แก้ไฟล์ที่ import เข้ามาใหม่ได้เลยโดยไม่ต้องย้อนดูบทสนทนาเดิม** (path เต็ม + field เต็ม + ค่าเดิม→ค่าใหม่ + เหตุผล/หลักฐานอ้างอิง)

คนละไฟล์กับ `README.md` (ซึ่งเป็น log สรุปสั้นๆ ว่าแก้อะไรไปแล้ว) — ไฟล์นี้คือ**สเปคละเอียดของการแก้แต่ละจุด** ที่ใช้ซ้ำได้

### 1. Grid master — dummy grid `3'` และ `3''`

**ไฟล์:** `json_แก้ไขแล้ว/01บ้าน_เล็ก_1ชั้น_01/บ้าน_เล็ก_1ชั้น_01_หน้า00_gridline.json`
**Field:** `grid.x_lines[]` (array ของ x-grid)

**ก่อนแก้:**
```json
{ "id": "3'", "pos_m": 8.5, "type": "dummy", ... }
```
(มีแค่เส้นเดียว)

**หลังแก้:**
```json
{ "id": "3'", "pos_m": 7.6, "type": "dummy", "confidence_score": 1, ... },
{ "id": "3''", "pos_m": 8.5, "type": "dummy", "confidence_score": 1, ... }
```

**เหตุผล/หลักฐาน:**
- `3'` (7.6 = grid3[7.0] + 0.60): หน้า05 (A-03 ผังบริเวณ, `element_id: "building_footprint"`) ระบุตรงๆ ว่าตัวอาคารยื่นเกินกริด 3 แค่ 0.60m — เป็นขอบผนัง/คานทับหลังจริง คอนเฟิร์มด้วย tie-beam corner projection 0.60m บนแปลนอะเส (หน้า20)
- `3''` (8.5 = grid3[7.0] + 1.50): หน้า19 (S-02 beam plan) คาน B4X วิ่งเต็ม 1.5m จาก C-3/B-3 — คือคานยื่นออกไปรับชายคา/กันสาดเลยขอบผนังจริง (3') ออกไปอีก คนละเส้นกับ 3' จริง
- **ค่าเก่า (8.5 เดี่ยวๆ) ผิด** เพราะเดิมอ้าง B4X เป็น "หลักฐานอิสระที่ 2" ยืนยัน 8.5 ทั้งที่จริง B4X's `span_source: "grid_table"` = คำนวณย้อนจาก grid master เอง (circular reference) ไม่ใช่วัดอิสระจากภาพ
- อัปเดต `dummy_grid_rule_check_2026-07-08.prime_ordering_rule_3.1.1` ด้วย ว่า `3'`/`3''` อยู่ช่องเดียวกัน (เหมือน A'/A'') เรียงตามระยะห่างจากกริด 3
- อัปเดต `non_grid_dimensions_do_not_confuse[]` entry ที่เคยบอกว่า "0.60 ไม่ใช่ 3'" → ตอนนี้คือจุดเดียวกันแล้ว

### 4. Convention: `confidence_score` สำหรับค่าที่มะขามยืนยันเอง

**กฎ:** ทุกครั้งที่มะขามสั่งแก้/ยืนยันค่าใดๆ ตรงๆ (ไม่ใช่ Claude เดา/ประมาณจากภาพ) → ตั้ง `confidence_score: 1` เสมอ พร้อมเพิ่ม `confidence_flags` อธิบายว่า "corrected/confirmed by user" + วันที่ + เหตุผลสั้นๆ

**ใช้กับ:** ข้อ 1, 2, 3 ข้างต้นทั้งหมด (และทุกไฟล์ที่แก้ต่อจากนี้)

### 5. `grid_ref` — ย่อ free-text ให้กระชับ

**ไฟล์:** `json_แก้ไขแล้ว/01บ้าน_เล็ก_1ชั้น_01/บ้าน_เล็ก_1ชั้น_01_หน้า06_floor_plan.json`
**Field:** `element[]` array, ทุก entry ที่มี field `grid_ref` เป็น string

**รูปแบบเก่า (ยาว, ใช้คำเต็ม):**
```
"grid1 - dummy1' x gridD - gridC"
"dummy1' - grid2 x gridD - gridC"
"near grid1 x gridC - gridB, see detail A-09/A-10"
"grid1'/2 - grid3 x gridC - gridB (approx)"
"beyond grid3 to dummy3' x gridD - gridC (open utility extension)"
"grid1 - approx mid x gridB - gridA (carport, open on one side)"
"approx mid - grid3 x gridB - gridA"
"grid2-3 area, below gridA (into dummy A' zone)"
"near grid1, between จอดรถ and เฉลียง — see detail A-11"
"near grid3/C row, between เตรียมอาหาร and ซักล้าง — see detail A-11"
```

**รูปแบบใหม่ (สั้น, ตัดคำว่า "grid"/"dummy" ออก, ใช้ `-` คั่นช่วง, `x` คั่นแกน, `~` แทน "approx/near", แกนตั้ง (y_lines, แถวตัวอักษร) มาก่อนเสมอ):**
```
"D-C x 1-2"
"D-C x 1'-2"
"C-B x ~1, see A-09/A-10"
"C-B x 1'/2-3 (approx)"
"D-C x 3-3' (open utility extension)"
"B-A x 1-~mid (carport, open one side)"
"B-A x ~mid-3"
"below A x 2-3 (into A' zone)"
"~1, between จอดรถ/เฉลียง — see A-11"
"C x ~3, between เตรียมอาหาร/ซักล้าง — see A-11"
```

**กฎทั่วไปสำหรับไฟล์อื่นที่ยังไม่ได้แก้ (ให้ใช้รูปแบบนี้ต่อไปทุกไฟล์ในบ้านนี้และบ้านอื่นถ้าเจอ pattern เดียวกัน) — เขียนไว้ใน `primary_rawjson_schema.md` section 4 ด้วยแล้ว:**
- ตัดคำว่า `grid`/`dummy` ออกจากหน้าเลข/ตัวอักษรกริดเสมอ (`grid1` → `1`, `dummy1'` → `1'`, `gridD` → `D`)
- ใช้ `x` คั่นระหว่างช่วงแกน — **แกนตั้ง (y_lines, แถวตัวอักษร A/B/C/D) มาก่อนเสมอ แล้วตามด้วยแกนนอน (x_lines, คอลัมน์ตัวเลข)** เช่น `"D-C x 1-2"` ไม่ใช่ `"1-2 x D-C"` — ตรงกับ convention เดิมของ `"A-1"` (row ก่อน column) อยู่แล้ว
- ใช้ `~` แทนคำว่า "near"/"approx" ที่ขึ้นต้น
- ข้อความอธิบายเพิ่มเติมที่จำเป็นจริง (เช่น "see detail A-09/A-10", "(open utility extension)") เก็บไว้ต่อท้ายในวงเล็บหรือหลัง comma ได้ ไม่ต้องตัดทิ้ง — ตัดแค่คำฟุ่มเฟือยของ grid notation เอง
- **แก้ไข 2026-07-13**: รอบแรกที่ย่อ เขียนแกนนอน (x) ก่อนโดยไม่ตั้งใจ (เช่น `"1-1' x D-C"`) มะขามสั่งแก้กลับเป็นแกนตั้งก่อนตามข้างบน — ถ้า import ไฟล์ใหม่มาแล้วเจอ `grid_ref` แบบยาว ให้ย่อลงมาเป็นแกนตั้งก่อนตั้งแต่รอบแรกเลย ไม่ต้องรอแก้ 2 รอบ

### 6. Grid master — เพิ่ม dummy grid `D'` (แกน y) + cascading fix ทุกหน้าที่เกี่ยวข้อง

**ไฟล์หลัก:** `json_แก้ไขแล้ว/01บ้าน_เล็ก_1ชั้น_01/บ้าน_เล็ก_1ชั้น_01_หน้า00_gridline.json`
**Field:** `grid.y_lines[]` — เพิ่ม entry ใหม่

**เพิ่มเข้ามา:**
```json
{ "id": "D'", "pos_m": 2.8, "type": "dummy", "confidence_score": 1, "confidence_flags": ["added_2026-07-13_by_user_confidence_set_to_1_per_user_verification", "NOT_YET_confirmed_by_beam_plan_หน้า19_which_only_has_beam_rows_at_D_C_B_A_no_beam_at_this_interior_line"] }
```

**เหตุผล/หลักฐาน:** หน้า06 (A-04 แปลนพื้นชั้น1) ช่วงหลัก D-C (4.00m) แบ่งย่อยเป็น 2.80+1.20 ตรงกับผนังกั้นเต็มความกว้างระหว่างแถวห้องนอน (นอน1/นอน2) กับแถวห้องด้านหลัง — มะขามสั่งเพิ่มตรงๆ

**⚠️ ข้อควรรู้ (ยังไม่ได้ยืนยันด้วยแปลนคาน):** เช็คแปลนคาน (หน้า19_view2_beam_plan.json) แล้วไม่มีคานแถวไหนอยู่ที่ตำแหน่งนี้เลย (มีแค่แถว D/C/B/A) — ต่างจาก `1'` ที่มีคาน B1 ยืนยัน เป็นไปได้ว่า `D'` คือผนังกั้นเบา (partition) ไม่ใช่เส้นคานโครงสร้างจริง — ใส่ flag ไว้ในไฟล์แล้ว ไม่ได้ลบทิ้งเพราะมะขามยืนยันให้เพิ่ม

**Cascading fix — ไฟล์ที่ต้องแก้ตามเมื่อเพิ่ม `D'`:**

1. **`หน้า06_floor_plan.json`** — นอน1/นอน2 เดิม `grid_ref` เป็น `"D-C x ..."` (เต็มช่วง D-C) ต้องแก้เป็น `"D-D' x ..."` เพราะห้องจริงสูงแค่ 2.80m ไม่ใช่เต็ม 4.00m:
   - นอน1: `"D-C x 1-2"` → `"D-D' x 1-2"`
   - นอน2: `"D-C x 2-3"` → `"D-D' x 2-3"`
   - **ห้องน้ำ_WC1: ตอนแรกจะแก้เป็น `"D'-C x ~1"` (คิดว่าห้องน้ำอยู่ในช่วงที่เหลือ D'-C) แต่เช็ค `หน้า11_wc1_plan.json` (แบบขยายห้องน้ำ WC1 โดยเฉพาะ) พบว่าระบุตรงๆ ว่าห้องอยู่ "between gridB and gridC" — คือ `C-B` เหมือนเดิม ไม่ใช่ `D'-C` ตามที่เดาไว้ — กลับไปใช้ `"C-B x ~1"` ตามเดิม (แค่แก้ format axis-order) — บทเรียน: ก่อน cascade แก้ grid_ref ตาม dummy grid ใหม่ ต้อง**เช็คไฟล์แบบขยาย/detail sheet ที่เกี่ยวข้องก่อนเสมอ** ถ้ามี เพราะอาจมีตัวเลขที่ยืนยัน/ขัดแย้งกับที่เดาไว้**
2. **`หน้า06_roof_plan.json`** — ไม่มี field ที่ต้องอ้าง D' โดยตรง แต่แก้ format `grid_ref` ทุก entry ให้ตรง convention (ตัด `grid`/`gridA` ออก, แกนตั้งก่อน) ไปพร้อมกัน เช่น `"covers grid1-3 x gridA-D..."` → `"D-A x 1-3..."`
3. **`หน้า11_wc1_plan.json`** — ไม่ต้องเปลี่ยนค่า (ถูกต้องอยู่แล้ว) แค่แก้ format: `"between gridB and gridC"` → `"C-B"`, ตัด `grid` prefix ออกจาก field อื่นๆ ในไฟล์เดียวกันด้วย
4. **`หน้า19_view2_beam_plan.json`, `หน้า20_view1_tie_beam_plan.json`** — เช็คแล้ว **ไม่ต้องแก้** เพราะคานจริงพาดเต็มช่วง D-C (ไม่มีคานหยุดที่ D') สอดคล้องกับที่ D' เป็นผนังกั้นไม่ใช่เส้นคาน

**กฎทั่วไป (เพิ่มเข้า convention):** ทุกครั้งที่เพิ่ม/แก้ dummy grid ใหม่ในกริดมาสเตอร์ ต้อง **ไล่เช็คทุกไฟล์ในบ้านเดียวกันที่มี `grid_ref` พาดผ่านช่วงนั้น** ว่าต้องแก้ตามไหม — ถ้ามีไฟล์แบบขยาย/detail sheet ของจุดเดียวกัน (เช่น WC1 มีทั้งหน้า06 และหน้า11) ให้เอาไฟล์ detail เป็นหลักฐานตัดสินก่อนเดาเอง

**เหตุผล:** มะขามสั่งเพิ่มตรงๆ — น่าจะมาจากผนัง/ขอบระหว่างโซนห้องน้ำ WC1 กับเตรียมอาหารบนหน้า06 (ใกล้ dimension "0.55" ที่พิมพ์ในภาพ) แต่ Claude ยังไม่ได้ pinpoint เส้น dimension ที่แน่ชัด 100% ด้วยตัวเอง — เพิ่มตาม prime-ordering (1' ที่ 2.55 ใกล้กว่า, 1'' ที่ 3.45 ไกลกว่า ในช่องเดียวกัน)

### 8. Convention ใหม่: point-type `grid_ref` ไม่ใช้ขีด (dash) — เก็บขีดไว้เฉพาะ line/area range

**กฎ:** การอ้างอิงกริดที่เป็น**จุดเดียว** (point — ตำแหน่ง footing/column, หรือปลายด้านหนึ่งของคาน) ให้เขียนติดกันไม่มีขีดคั่นระหว่างแถวตัวอักษรกับคอลัมน์ตัวเลข เช่น `"C2"` ไม่ใช่ `"C-2"` — ขีดคั่นสงวนไว้ใช้เฉพาะตอนอ้างอิง**เส้น/พื้นที่**ที่เป็นช่วงระหว่าง 2 ตำแหน่งบนแกนเดียวกันจริงๆ เช่น `"D-C"` (ช่วง D ถึง C) หรือ `"1-2"` (ช่วงกริด1 ถึงกริด2)

**เหตุผลจากมะขาม:** "เป็นแค่จุดๆ ไม่ใช่เส้น หรือพื้นที่ต้องมีขีด"

**ไฟล์ที่แก้แล้ว:**
1. **`หน้า19_view1_footing_plan.json`** — `grid_refs[]` ของ F1/F2/C1 (footing/column) จาก `["D-1","D-2",...]` → `["D1","D2",...]`
2. **`หน้า19_view2_beam_plan.json`** — `grid_ref_start`/`grid_ref_end` ของทุกคาน (B2/B4/B5/B3/B4X/B1/B3X/B5X) จาก `"D-1"` → `"D1"` ทั้งหมด (แต่ละอันคือจุดปลายคานจุดเดียว ไม่ใช่ตัวช่วงเอง — ช่วง/เส้นคานแสดงด้วยการมี 2 field แยก start/end อยู่แล้ว ไม่ต้องมีขีดซ้ำในตัวจุดเอง) ค่าที่มี prime ต่อท้าย (`C-1'` → `C1'`, `C-3''` → `C3''`) เก็บ prime ไว้ตามเดิม ตัดแค่ขีดคั่นแถว-คอลัมน์

4. **งานใหญ่สุดที่ยังไม่เริ่มเลย**: sync ผลการแก้ทั้งหมดใน `json_แก้ไขแล้ว/01บ้าน_เล็ก_1ชั้น_01/` กลับไปทับ `raw_json_ตัวที่ใช้งานจริง/01บ้าน_เล็ก_1ชั้น_01/` ต้นฉบับ (ตามกฎข้อ 1-3 rule_of_tune.md เต็มรูปแบบ) — ยังไม่ได้ตกลงว่าจะ sync ตอนไหน
5. **บ้านหลังถัดไป (หลังที่ 6) ยังไม่เริ่มรีวิว** — ถ้าจะเริ่ม ควรสร้าง `json_แก้ไขแล้ว/06<ชื่อบ้าน>/` ก่อน (คัดลอกจาก raw) แล้วเริ่มไล่ตาม convention ที่ยืนยันแล้วในบ้าน 1 (ตัดขีด point-ref, แกนตั้งก่อน, element ordering แนวตั้งก่อน, specs{} แยก spec ซ้ำ, ตรวจ additional_bars position กับ leader line เสมอ, เสาไม่ใช้ top/bottom)
