### 5. `grid_ref` — shorten free text to be concise

**File:** `json_แก้ไขแล้ว/01บ้าน_เล็ก_1ชั้น_01/บ้าน_เล็ก_1ชั้น_01_หน้า06_floor_plan.json`
**Field:** `element[]` array, every entry with a string `grid_ref` field

**Old format (long, full words):**
```
"grid1 - dummy1' x gridD - gridC"
"dummy1' - grid2 x gridD - gridC"
"near grid1 x gridC - gridB, see detail A-09/A-10"
"grid1'/2 - grid3 x gridC - gridB (approx)"
"beyond grid3 to dummy3' x gridD - gridC (open utility extension)"
"grid1 - approx mid x gridB - gridA (carport, open on one side)"
"approx mid - grid3 x gridB - gridA"
"grid2-3 area, below gridA (into dummy A' zone)"
"near grid1, between จอดรถ and เฉลียง — see detail A-11"
"near grid3/C row, between เตรียมอาหาร and ซักล้าง — see detail A-11"
```

**New format (short, drops "grid"/"dummy", uses `-` for ranges, `x` between axes, `~` for "approx/near", vertical axis (y_lines, letter rows) always comes first):**
```
"D-C x 1-2"
"D-C x 1'-2"
"C-B x ~1, see A-09/A-10"
"C-B x 1'/2-3 (approx)"
"D-C x 3-3' (open utility extension)"
"B-A x 1-~mid (carport, open one side)"
"B-A x ~mid-3"
"below A x 2-3 (into A' zone)"
"~1, between จอดรถ/เฉลียง — see A-11"
"C x ~3, between เตรียมอาหาร/ซักล้าง — see A-11"
```

**General rule for other not-yet-fixed files (use this format going forward for every file in this house, and other houses if the same pattern appears) — also written into `primary_rawjson_schema.md` section 4:**
- Always strip `grid`/`dummy` from grid letters/numbers (`grid1` → `1`, `dummy1'` → `1'`, `gridD` → `D`)
- Use `x` between the two axis ranges — **the vertical axis (y_lines, letter rows A/B/C/D) always comes first, followed by the horizontal axis (x_lines, numbered columns)**, e.g. `"D-C x 1-2"` not `"1-2 x D-C"` — matches the existing convention for `"A-1"` (row before column)
- Use `~` in place of a leading "near"/"approx"
- Genuinely necessary extra description (e.g. "see detail A-09/A-10", "(open utility extension)") can stay appended in parentheses or after a comma — don't strip it, only strip the redundant grid-notation wording itself
- **Fixed 2026-07-13**: the first shortening pass unintentionally wrote the horizontal axis (x) first (e.g. `"1-1' x D-C"`); Makham had it corrected back to vertical-axis-first per above — if a newly-imported file has a long-form `grid_ref`, shorten it with vertical-axis-first from the very first pass, don't wait for a second correction round.

### 8. New convention: point-type `grid_ref` uses no dash — dash reserved for line/area ranges only

**Rule:** a grid reference that is a **single point** (a footing/column position, or one end of a beam) should be written with no dash between the letter row and the number column, e.g. `"C2"` not `"C-2"` — the dash is reserved specifically for referencing a **line/area**, a range between 2 positions on the actual same axis, e.g. `"D-C"` (the range from D to C) or `"1-2"` (the range from grid 1 to grid 2).

**Makham's reasoning:** "it's just a point, not a line — only a line or area needs a dash."

**Files already fixed:**
1. **`หน้า19_view1_footing_plan.json`** — `grid_refs[]` for F1/F2/C1 (footing/column) changed from `["D-1","D-2",...]` → `["D1","D2",...]`

Values with a trailing prime (`C-1'` → `C1'`, `C-3''` → `C3''`) keep the prime as-is, only the row-column dash is stripped.

**⚠️ Not yet applied project-wide** — already asked Makham whether this should become a new convention for the whole project (every file, every house) or just house 1, currently being reviewed; still awaiting a clear answer, and `primary_rawjson_schema.md` section 4 (which currently says the opposite, that point-type uses `"A-1"` with a dash) has not been updated yet — must wait for confirmation first, per rule_of_tune.md rules 7/9.
